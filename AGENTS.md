# AGENTS.md

이 저장소의 기본 작업 방식은 다음과 같습니다.

## 이 저장소 (GalleryCRM)에 대해

- **실행·패키지**: 백엔드는 **npm + NestJS + Jest + Node + `pg`**, 프론트는 **npm + Create React App(`react-scripts`)**. 구체는 저장소 루트 `README.md`와 `.cursor/rules/20-gallerycrm-tooling.mdc`를 따릅니다.
- **제품 형태**: **웹**(React SPA + Nest API)이 기본입니다. 별도 네이티브 앱이 프로젝트 범위에 없으면, 아래 "앱의 목록 화면" 등 **모바일 앱 UX** 항목은 참고만 하고 강제하지 않습니다.
- **문서 역할**: 기존 `docs/PROJECT_SUMMARY.md` 등과 워크플로 산출물(`docs/prd.md` 등)의 관계는 **[docs/DOCUMENTATION_MAP.md](./docs/DOCUMENTATION_MAP.md)**에 정리되어 있습니다.
- **규칙의 우선순위**: 절차·산출물 경로·게이트는 이 파일과 **`.cursor/rules/00-workflow.mdc`**가 같은 방향입니다. 세부 운영 규칙(blocker, hooks, worktree 등)은 **워크플로 규칙 파일**을 우선합니다.

## 기본 원칙
- 사용자가 명시적으로 바로 구현하라고 하지 않는 한, 먼저 계획부터 세운다.
- 작업을 시작할 때 현재 요청을 하위 작업으로 나누어 본다.
- 사용자가 별도로 제외하지 않는 한, 사용자 노출 기능은 한국어/영어 2개 언어 지원 가능성을 기본 요구사항으로 검토한다.
- 각 하위 작업에 적합한 subagent가 있는지 먼저 확인한다.
- 적합한 subagent가 있으면 해당 단계는 우선 위임한다.
- 적합한 subagent가 없거나 작업이 매우 단순하면 메인 에이전트가 직접 처리할 수 있다.
- 문서가 비어 있거나 오래되었으면 먼저 업데이트한다.
- 구현 후에는 반드시 테스트, 리뷰, blocker 수정, 게이트 확인, 문서 업데이트까지 진행한다.
- 중간 보고는 아래 경우에만 한다:
  - blocker 발생
  - 범위 변경 필요
  - 중요한 의사결정 필요
- 그 외에는 end-to-end로 진행한다.

## 기본 작업 순서
1. research
2. prd
3. frontend/backend planning
4. ui spec
5. implementation
6. testing
7. review
8. blocker fix if needed
9. re-test
10. gate check
11. docs update

## subagent 사용 기준
- 먼저 현재 작업을 하위 작업으로 나눈다.
- 각 하위 작업과 유사한 역할의 subagent가 있는지 확인한다.
- 적합한 subagent가 있으면 우선 해당 작업을 위임한다.
- 적합한 subagent가 없으면 메인 에이전트가 직접 처리한다.
- 작업이 매우 작고 단순해서 위임 비용이 더 크다고 판단되면 직접 처리할 수 있다.
- subagent를 사용하지 않은 경우, 완료 보고에서 간단한 이유를 남긴다.

## subagent 매핑 기준
- 도메인/기존 구조 조사 -> research-agent
- 요구사항/범위/MVP 정리 -> prd-agent
- UI 구현 -> frontend-agent
- API/스키마/서비스 구현 -> backend-agent
- 테스트/검증 -> tester-agent
- 품질/회귀/UX 리뷰 -> review-agent
- 문서 정리 -> docs-agent

## 응답 방식
- 평소에는 길게 설명하지 말고 작업을 진행한다.
- blocker, 범위 변경, 중요한 선택지만 짧게 보고한다.
- 완료 시에는 아래를 요약한다:
  - 무엇을 했는지
  - 무엇을 검증했는지
  - 남은 리스크가 있는지
  - 사용한 subagent
  - 직접 처리한 작업
  - subagent를 사용하지 않은 이유

## 계획 우선 원칙
- 먼저 현재 코드베이스와 관련 문서를 조사한다.
- 구현 전 필요한 문서를 먼저 생성하거나 갱신한다.
- 문서가 준비되기 전에는 구현하지 않는다.
- 구현은 승인된 문서와 계획을 기준으로 진행한다.

## 테스트와 게이트
- 구현 후 tester-agent를 통해 검증한다.
- 검증 후 review-agent를 통해 blocker와 회귀 위험을 점검한다.
- blocker가 있으면 blocker만 수정하고 재검증한다.
- 최종 완료 전 docs/gate-checklist.md를 기준으로 게이트를 확인한다.

## UI / UX 원칙
- 사용자에게 보이는 라벨, 타이틀, 버튼 문구, 얼럿, 토스트, validation 메시지를 자연스럽고 일관되게 유지한다.
- 성공, 실패, 로딩, 빈 상태, 에러 상태를 명시적으로 확인한다.
- 데스크톱과 모바일 사용감을 함께 고려한다.
- 사용자가 별도로 단일 언어만 요청하지 않는 한, 새로 만드는 앱/웹은 한국어와 영어를 함께 지원할 수 있도록 구조를 고려한다.
- 초기 MVP에서 다국어 리소스를 모두 완성하지 못하더라도, 사용자 노출 텍스트는 하드코딩을 최소화하고 추후 i18n 적용이 가능하도록 정리한다.
- 라벨, 버튼, 타이틀, 얼럿, 토스트, validation 메시지 등 사용자 노출 문구는 한국어 기준으로 먼저 자연스럽게 작성하고, 영어 버전도 함께 정의할 수 있는 구조를 우선한다.
- 언어 전환 기능이 범위에 포함되는 경우, 데스크톱과 모바일에서 모두 자연스럽게 동작하도록 고려한다.

- 웹에서 테이블이 있는 화면은 특별한 이유가 없으면 상단에 검색 또는 필터 영역을 둔다.
- 웹의 테이블 기반 목록 화면은 특별한 요구사항이 없으면 한 페이지에 15개 항목이 보이도록 페이지네이션을 적용한다.
- 웹의 페이지네이션은 테이블 하단 중앙에 가로로 배치한다.
- 웹의 페이지네이션에서 이전/다음 버튼은 각각 꺽새(<, >) 형태의 아이콘 또는 기호로 표현한다.
- 웹의 테이블 화면에서는 현재 페이지, 이동 가능 여부, 비활성화 상태가 명확하게 보이도록 구성한다.
- 웹의 테이블 화면에서는 필터, 로딩 상태, 빈 상태, 결과 없음 상태를 함께 고려한다.

- 앱의 목록 화면은 특별한 이유가 없으면 페이지네이션보다 인피니티 스크롤 방식을 우선한다.
- 앱의 목록 화면에서는 추가 데이터 로딩 중 상태, 마지막 데이터 도달 상태, 빈 상태를 명확하게 제공한다.
- 앱에서 필터가 필요한 목록 화면은 상단 고정 영역, 시트, 모달 등 모바일 사용성에 맞는 방식으로 제공한다.

- 프론트엔드 작업 전 전역 스타일 기준 파일(index.css)의 존재 여부를 먼저 확인한다.
- index.css가 필요한 프로젝트인데 제공되지 않았거나 저장소에 없으면, 임의의 전역 스타일 기준을 만들기 전에 사용자에게 먼저 요청한다.
- 사용자가 별도로 제외하지 않는 한, 앱/웹은 다크모드를 기본 지원 대상으로 고려한다.
- 다크모드는 단순 배경색 변경이 아니라 텍스트 대비, 보더, 입력창, 버튼, 카드, 모달, 토스트, 빈 상태, 에러 상태까지 일관되게 적용되도록 한다.
- 라이트 모드와 다크 모드 모두에서 가독성과 시각적 위계를 유지한다.

## 문서화 원칙
- 새 기능, 실행 방법, 검증 방법, 제한사항이 바뀌면 관련 문서를 업데이트한다.
- 사용자나 개발자가 실제로 참고할 가능성이 있는 변경이면 README.md 또는 관련 docs를 업데이트한다.
- 실제로 구현되지 않은 내용은 문서에 쓰지 않는다.
