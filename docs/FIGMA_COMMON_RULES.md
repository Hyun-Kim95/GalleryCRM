## 0. 공통 원칙

- **우선순위: 실제 서비스 화면 > 피그마**
  - 이미 개발 완료된 실제 서비스 화면이 **단 하나의 기준(소스 오브 트루스)** 이다.
  - 피그마는 이 기준을 **정확히 복제해서 고객사에 제출하기 위한 산출물**이다.
  - 따라서, 피그마를 수정할 때는 “디자인 아이디어”가 아니라 **실제 화면과의 차이만 줄이는 작업**만 한다.

- **코드 편집 금지**
  - 이 규칙은 **피그마 작업 전용**이다.
  - 실제 서비스 코드(프론트/백엔드/스타일)는 이 규칙과 무관하며, 피그마를 맞추기 위해 코드를 수정하지 않는다.

- **로컬 좌표 기준 작업**
  - 항상 **현재 부모 프레임 기준의 x, y(로컬 좌표)**로 요소를 배치한다.
  - 툴(Figma API 등)이 보여주는 `absoluteBoundingBox.x / y`는 **겉보기엔 절대좌표처럼 보여도,  
    실제로는 `부모 프레임의 위치 + 로컬 좌표`를 더한 값**이라는 점을 항상 기억한다.
  - 위치를 수정할 때는:
    - 먼저 **부모 프레임 기준에서 원하는 로컬 좌표(예: x=16, y=20)** 를 결정하고,
    - **툴에 넣는 값도 항상 그 “로컬 좌표 그대로” 넣는다.**  
      - 예: `Customer Detail Main` 의 로컬 (16, 146)에 카드를 두고 싶으면, 툴에 `x=16, y=146` 을 넣는다.  
      - **`부모 absolute + 로컬` 을 툴에서 다시 계산해서 넣지 않는다.** 그렇게 하면 로컬 좌표가 수천 단위가 되어 프레임 밖으로 날아간다.
    - 디자이너가 Figma UI에서 보는 로컬 좌표와, 툴로 입력한 값이 **완전히 동일**해야 한다.
  - 단, **디자인 리뷰/작업 시 기준은 항상 “Figma UI에 보이는 로컬 좌표 값”** 이다.  
    - 디자이너가 `x=16, y=146`이라고 말하면, 툴을 사용할 때도 **해당 노드의 로컬 좌표를 정확히 그 값으로 맞춘다.**
    - 부모 프레임 자체의 위치를 움직여 문제를 해결하려 하지 말고, **자식 노드의 로컬 좌표를 직접 설정**하는 것을 우선으로 한다.
  - 부모 프레임가 잘못되면, 먼저 **부모를 올바르게 설정**한 뒤 좌표를 조정한다.

- **“문서 → 실제 화면 → 피그마” 순서**
  1) 이 공통 규칙 문서를 읽는다.
  2) 실제 서비스 화면을 확인한다.
  3) 피그마를 실제 화면과 비교하면서, **차이 나는 부분만** 규칙에 맞춰 수정한다.
  4) 문서에 정의되지 않은 변경이 필요하면, 먼저 문서에 규칙을 추가/수정한 뒤 그에 따라 작업한다.

## 1. 화면 정보 정리 규칙

- 각 화면에서 작업을 시작하기 전에, 다음 정보를 간단히 적어 둔다.
  - 화면 이름, 피그마 프레임 이름, 실제 서비스 URL/경로
  - “이 피그마는 어떤 실제 화면을 그대로 복제하는가?”에 대한 한 줄 설명
- **프레임 이름(번호) 규칙**
  - 모든 상위 화면 프레임은 **번호 + 점 + 화면명** 형식으로 일관되게 붙인다.  
    - 예: `04. Customers`, `05. Customer Detail`, `06. Customer Create`, `07. Artists`, `08. Artist Detail`
  - **새 화면 프레임을 만들 때는 항상 `create_frame` 단계에서 올바른 이름을 넣어서 생성한다.**
    - 예: 07번 오른쪽/아래에 새 화면을 만들면, `name: "08. Artist Detail"` 처럼 **목표 번호를 포함한 이름**으로 바로 만든다.
  - **기존 프레임을 `clone_node` 로 복제해서 새 화면을 만들지 않는다.**
    - 클론을 쓰면 이름이 `05. Customer Detail` 처럼 이전 이름 그대로 복제되어,  
      나중에 번호가 뒤섞이고 화면 구조를 파악하기 어렵다.
    - 불가피하게 클론을 썼다면, **반드시 디자이너가 피그마 UI에서 프레임 이름을 즉시 올바른 번호/이름으로 수정한다.**
  - 툴 작업자는 “이번에 만드는 화면이 **몇 번 프레임인지**” 를 항상 먼저 확인하고,  
    **그 번호를 포함한 정확한 이름으로 생성**했는지 체크리스트 수준으로 확인한다.
- 화면별 상세 규칙(좌표, 크기, 간격 등 숫자)은 **별도 문서나 주석으로 관리**하되,  
  이 공통 규칙을 항상 함께 참조한다.

## 2. 영역별 공통 규칙

### 2-0. 공통 스타일 규칙

- **버튼 및 인풋 테두리**
  - 모든 버튼(button)과 인풋(input, select, textarea) 요소는 **둥근 테두리(cornerRadius: 6px)**를 적용한다.
  - 이 규칙은 모든 화면의 모든 버튼과 인풋에 일관되게 적용되며, 실제 서비스 화면에서도 동일한 스타일이 적용되어 있다.
  - 예외: 상태 배지(badge)는 별도 규칙에 따라 `cornerRadius: 12px`를 사용할 수 있다.

### 2-1. 검색/필터 영역

- **역할**
  - 실제 화면에서 “검색/필터”로 쓰이는 요소(검색 인풋, 셀렉트, 버튼 등)를 그대로 복제한다.

- **요소 정의**
  - 실제 화면을 기준으로, 어떤 필드/버튼이 있는지 **이름과 역할**을 명확히 적어 둔다.

- **레이아웃 규칙(정성적)**
  - 실제 화면의 정렬 방향(세로/가로), 순서, 간격을 그대로 따른다.
  - 모든 요소는 **검색 카드(부모 프레임)** 안에 완전히 포함되도록 배치한다.
  - 좌우 여백은 카드 양 끝에 너무 붙지 않게, **일관된 여백**을 유지한다.
  - 실제 코드가 `width: 100%` + `min-width` 조합을 쓰는 경우:
    - 피그마에서는 **카드 안에서 가로를 거의 꽉 채우는 폭(=100% 느낌)** 으로 맞춘다.
    - 예: `min-width: 150px` 정도인 셀렉트가 단일 필터로 한 줄을 차지한다면,
      - 데스크톱 기준으로는 **검색 카드의 내부 폭 - 좌우 여백(예: 16px / 16px)** 만큼을 필터 폭으로 잡는다.
      - 모바일/태블릿 레이아웃은 실제 서비스에서 컬럼 숨김/줄바꿈이 어떻게 되는지 보고 그대로만 복제한다.

- **텍스트 규칙**
  - 플레이스홀더, 버튼 레이블 등은 **실제 화면의 텍스트와 1:1로 동일**하게 쓴다.
  - 텍스트를 바꾸고 싶다면, 먼저 실제 서비스/기획에서 변경된 것이 맞는지 확인한다.

> 수정 시 원칙
> - 새로운 입력 요소를 임의로 추가하지 않는다.
> - 요소를 카드 밖으로 보내 숨기지 않는다. 안 보이면 부모/좌표부터 확인한다.

---

### 2-2. 요약/메타 정보 영역

- **역할**
  - 실제 화면에서 카드 위/아래/옆에 있는 “설명, 카운트, 상태 요약, 레이블, 액션 영역”을 복제하는 용도이다.

- **위치 규칙**
  - 이 영역이 실제 화면에서 검색/필터, 테이블과 어떤 상대 위치(위/아래/옆)에 있는지 텍스트로 먼저 정의한다.
  - 피그마에서도 이 상대 위치 관계를 그대로 유지한다.

- **스타일/간격 규칙**
  - 아이콘/텍스트/버튼의 배치는 실제 화면과 최대한 동일하게 맞춘다.
  - 새로운 UI 해석을 하지 않고, **보이는 그대로**만 옮긴다.

> 수정 시 원칙
> - 이 영역이 “요약/보조 정보” 역할이면, 검색/테이블 요소와 성격을 섞지 않는다.
> - 탭/필터 같은 동작을 추가하고 싶다면, 기획/문서를 먼저 수정한 뒤 반영한다.

---

### 2-3. 테이블 영역

- **역할**
  - 실제 서비스의 리스트(테이블) 화면을 **컬럼/정렬/텍스트까지 동일하게 복제**한다.

- **헤더 규칙**
  - 실제 화면의 컬럼명(예: 이름, 이메일, 상태 등)을 그대로 사용한다.
  - 헤더 텍스트와 데이터 열의 정렬 방식(좌/우/가운데)을 그대로 따른다.

- **행/컬럼 정렬 규칙**
  - 각 행의 세로 간격은 실제 화면과 동일하게 보이도록 맞춘다.
  - 각 컬럼의 가로 위치는 헤더 기준으로, **모든 행에서 동일한 x 라인**을 유지한다.
  - 상태 배지, 아이콘, 버튼 등도 실제 화면의 위치/정렬을 그대로 따른다.

> 수정 시 원칙
> - 행을 추가/삭제할 때, 세로 간격과 각 컬럼의 가로 위치가 모두 동일하게 유지되도록 한다.
> - 텍스트가 안 보이면, 부모 프레임, 좌표, visible 상태를 차례로 점검한다.

#### 2-3-1. 컬럼 간격/비율 맞추기 (데스크톱 기준)

- **기준 화면**
  - 항상 **데스크톱 레이아웃(최대 폭, 브라우저 넓은 상태)** 을 기준으로 숫자를 잡는다.
  - 모바일/태블릿에서 컬럼이 숨겨지거나 줄바꿈되는 것은, 데스크톱 기준을 먼저 맞춘 뒤 그대로 따라간다.

- **컬럼 x 좌표를 잡는 순서**
  1. 실제 서비스에서 해당 테이블과 **가장 유사한 기 구현 화면(예: Customers 리스트)** 을 찾는다.
  2. 그 화면의 피그마 테이블에서 **각 컬럼 헤더의 x 좌표 간 간격(픽셀 차이)** 를 기록한다.
     - 예: `고객(이름) → 작가` 사이 110px, `작가 → 금액` 사이 140px … 처럼 상대 간격을 적어 둔다.
  3. 새 테이블(예: Transactions)에서는:
     - **첫 번째 컬럼(고객/이름 등) x 를 기준점** 으로 두고,
     - 나머지 컬럼들을 **이미 기록한 간격 비율**을 최대한 유지하면서 오른쪽으로 배치한다.
  4. 마지막 컬럼(예: `작업`)은:
     - 오른쪽 여백이 과하게 남지 않도록, **직전 컬럼과의 간격이 다른 컬럼 간격과 크게 어긋나지 않는 수준** 으로 조정한다.
     - 버튼/아이콘 텍스트가 잘리지 않을 최소 폭을 확보한 뒤, 남는 여백이 있으면 **테이블 전체 폭 안에서 균형 있게 분배**한다.

- **상태/날짜/작업 같은 좁은 컬럼 처리**
  - `상태` 배지, 짧은 날짜(`2026. 02. 03.`), `작업` 버튼은:
    - 텍스트/배지/버튼이 한 줄에 자연스럽게 들어가도록 최소 폭을 먼저 계산한다.
    - 그 다음, **서로 겹치지 않을 정도로만 간격을 벌리고**, 나머지 가용 폭은 긴 텍스트(이름/제목/금액 등)에 우선 배정한다.
  - `작업` 컬럼은:
    - 버튼 하나가 들어갈 정도의 폭만 주고, 오른쪽 여백이 지나치게 넓지 않도록 **등록일 바로 옆 정도 위치**에 둔다.
    - 실제 구현 코드에서 `table` 이 자동으로 칼럼 폭을 나누는 모습을 참고해, **눈으로 봤을 때 비슷한 비율**이 되도록만 맞춘다.

## 3. 금지/주의 사항

- **디자인 “창조” 금지**
  - 이 작업의 목적은 **새 디자인을 만드는 것이 아니라, 이미 있는 실제 화면을 복제하는 것**이다.
  - 이미 구현된 서비스와 다른 레이아웃, 텍스트, 컬럼을 임의로 추가하거나 변경하지 않는다.

- **코드 기준으로 피그마를 바꾸지 않음**
  - “이렇게 하면 개발이 쉬울 것 같다”는 이유로 피그마를 바꾸지 않는다.
  - 기준은 언제나 실제 서비스 화면의 결과물이다.

- **프레임 밖 숨기기 금지**
  - 요소를 안 보이게 하려고 프레임 밖 멀리 보내지 않는다.
  - 숨김이 필요하면 visible 속성이나 별도 레이어 상태로 관리하고, 좌표는 유지한다.

- **문서 동기화 필수**
  - 실제 화면이 변경되면, 피그마를 바꾸기 전에 **먼저 이 규칙이나 별도 문서를 갱신**하고,  
    그 다음 피그마를 다시 실제 화면 기준으로 맞춘다.
