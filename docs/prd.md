# PRD — 제품 요구사항 (GalleryCRM / Prism)

> `PROJECT_SUMMARY.md`는 구현 이력 스냅샷입니다. **제품이 충족해야 할 요구**는 본 문서를 기준으로 합니다. 작업 단위 기능은 섹션 6·7과 별도 이슈로 추적합니다.

## 1. 배경·목표

- **문제 / 기회**: 갤러리 내부에서 고객·작가·거래 정보를 역할에 맞게 안전하게 등록·조회·승인하고, 민감 데이터는 마스킹·열람 승인 절차로 보호해야 한다.
- **성공 시 모습**: 사내 사용자가 웹 브라우저에서 로그인 후, 권한에 맞는 데이터만 자연스럽게 조회·등록하고, 승인·감사 추적이 끊기지 않는다.

## 2. 범위

### 포함 (In scope)

- 웹 클라이언트(React SPA) + REST API(NestJS) + PostgreSQL(TypeORM)
- JWT 기반 로그인·세션(토큰) 및 역할 기반 접근 제어(RBAC)
- 팀 단위 데이터 경계 및 데이터 마스킹(역할·소속에 따른 PARTIAL/FULL)
- 마스킹 데이터 **열람 요청 → 승인/거부 → 시간 제한 열람** 흐름
- 주요 엔티티 **등록/수정 승인 워크플로**(Draft → Pending → Approved/Rejected 등, 도메인 정의 준수)
- 검색·필터·목록·상세 조회(페이지네이션 등 UI 패턴은 `AGENTS.md` / UI 명세와 정합)
- 감사 로그(Audit Log) 및 엔티티 변경 이력(Entity History) 기록·조회
- 관리자 중심 사용자·팀·승인·감사 기능(화면·API는 코드베이스 및 `API_SUMMARY.md`와 일치)

### 제외 (Out of scope) — 별도 결정 전까지

- 공개 인터넷 멀티테넌트 SaaS화, 고객(외부) 셀프 서비스 포털
- 네이티브 모바일 앱(요구 시 별도 PRD)
- 필드 단위 암호화 저장·별도 KMS 연동(보안 강화 백로그로 관리 가능)
- 실시간 알림 인프라(이메일/푸시 등) — 제품 로드맵 항목

## 3. 사용자·권한

- **역할**: `ADMIN`, `MANAGER`, `STAFF`(실제 enum·표기는 코드베이스 준수)
- **데이터 접근 원칙**:
  - 본인/소속 팀 vs 타 팀 데이터 구분
  - **ADMIN**: 전역 운영·승인 권한(도메인 규칙 범위 내)
  - **MANAGER**: 소속 팀 중심 데이터 및 승인 권한(도메인 규칙 범위 내)
  - **STAFF**: 제한된 열람·등록; 마스킹 규칙 준수
- **마스킹**: 이름·연락처·금액 등 민감 필드는 마스킹 수준(PARTIAL/FULL)이 역할·팀 정책과 일치해야 한다.
- **열람 승인**: 마스킹된 정보가 필요할 때 열람 요청·승인 없이 우회 열람이 되면 안 된다.

## 4. 기능 요구사항

| ID | 요구사항 | 수용 기준 (검증 방법) |
|----|----------|------------------------|
| F-1 | 사용자는 이메일·비밀번호로 로그인할 수 있다. | 유효 계정으로 로그인 성공; 잘못된 자격 증명 시 인증 실패; 보호 라우트는 비로그인 접근 불가 |
| F-2 | 역할·팀에 따라 API·화면에서 허용된 작업만 수행할 수 있다. | 역할별로 금지 작업 시 403 등 일관된 거부; 클라이언트에서도 권한 없는 내비게이션/액션 제한 |
| F-3 | 민감 필드는 마스킹 규칙에 따라 표시된다. | PARTIAL/FULL 규칙이 백엔드 응답·UI에 반영; 정책 위반 노출이 없어야 함 |
| F-4 | 마스킹 데이터 열람이 필요하면 열람 요청을 생성할 수 있고, 승인자는 승인/거부할 수 있다. | 요청 생성 → 승인/거부 → 승인 시에만 제한적 열람 가능; 거부 시 사유 등 도메인 규칙 반영 |
| F-5 | 고객·작가·거래 등 핵심 엔티티는 등록·수정 시 승인 워크플로를 따른다. | Draft/Pending/Approved/Rejected 등 상태 전이와 승인 전 제한이 API·UI에서 일치 |
| F-6 | 목록·검색·필터·페이지네이션으로 업무 목록을 조회할 수 있다. | 주요 목록 화면에서 검색/필터/페이지 이동이 동작; 빈 결과·로딩 상태 처리 |
| F-7 | 중요 행위는 Audit Log에 남고, 필요 시 Entity History로 변경을 추적할 수 있다. | 감사 로그 API·화면에서 조회 가능; 변경 이력 요구가 있는 엔티티는 추적 가능 |
| F-8 | 관리자 기능(사용자·팀·승인·감사 등)이 요구 수준으로 제공된다. | `API_SUMMARY.md`·실제 라우트와 기능이 대응; 문서와 불일치 시 문서 또는 구현을 정합 |

## 5. 비기능·제약

- **보안**: 프로덕션에서는 `JWT_SECRET`·DB 자격 증명 등 환경 변수를 기본값 그대로 쓰지 않는다. HTTPS 배포는 운영 정책에 따른다.
- **감사**: 감사 로그는 책임 추적 가능하도록 사용자·엔티티·액션 식별이 가능해야 한다.
- **호환**: 최신 브라우저(Chrome/Firefox/Safari 최근 버전)에서 핵심 플로우 동작.
- **성능**: MVP 수준에서 목록·상세 응답이 업무에 지장 없을 정도(구체 SLA는 운영 단계에서 정의).
- **다국어**: 사용자 노출 문구는 한국어 우선; 구조적으로 i18n 확장 가능하게 유지(`AGENTS.md`).
